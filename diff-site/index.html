The core defect

Approximately half of recorded wins are not sales

The published workflow 05. Clinic Appt Outcome creates a Won opportunity in the A&D ("Approved & Declined") pipeline for every member who attends a consultation, independent of purchase outcome. As a result, win-rate and revenue reporting materially overstate performance.

A&D holds 939 won / 0 lost — approximately 50% of all 1,864 recorded wins account-wide. Across sampled won deals, 57–99% carry $0 monetary value: revenue lives in contact fields that are mostly empty, and GHL has zero payment transactions.

      Node-level evidence

      Root cause: the configuration generating misclassified wins

      Captured from the GHL workflow builder. This is one Create Opportunity action inside the SHOWED branch of 05. Clinic Appt Outcome.

      ⚙︎ Create Opportunity — action config
      Action nameA&D pipeline: Closed, status: WON
      In pipeline07. A & D
      In stageClosed
      Opportunity name{{contact.name}} SHOWED
      Opportunity value{{contact.total_program_amount}}
      StatusWON

      ⚠ Why this corrupts reporting

      The trigger is SHOWED — attendance, not purchase. Every member who attends a consultation is recorded as a Won opportunity, regardless of purchase outcome.

      The value reads from the contact field total_program_amount, which is empty for most contacts — so the majority of these records carry a $0 value.

      Win rate, revenue, and by-location performance are all computed on top of this. Correcting this will reduce the reported win rate by approximately half. This restatement reflects accurate measurement of historical performance, not a decline in actual performance.

      How each workflow gets built — browser method

      1. Fresh browser session per workflow — the GHL builder is heavy; one workflow per tab, builder opened from the Workflows list (direct URLs hang).
      2. Open the source workflows in a second tab — copy trigger filters, tag names, and message bodies verbatim where they're being kept; the engine's existing tag IDs are reused, never recreated.
      3. Build the new workflow as a Draft — draft workflows do not execute in GHL, so production is unaffected until Publish.
      4. Verify against this spec — every trigger, branch, wait, and message below is checked off on the canvas before the workflow is marked build-complete.
      5. Test contact end-to-end — drive a test contact through the draft (Test Workflow), assert tags, fields, opportunity moves, and message rendering.
      6. Cutover behind the canary — publish the new workflow, pause (never delete) the sources it absorbs, watch 48h, then widen.
      7. Published-only accounting — 36 of the 38 published workflows map into the 16 below; the remaining 2: 06. Price Calculator stays standalone, and New Workflow : 1780243709697 (unnamed but published) gets inspected, then deleted.

      The test matrix

      Every scenario asserted via API by Claude — create the test contact, drive the event, assert the resulting state. Engine scenarios run on rehearsal day.

      Never: workflow-moves opps

      Cross-pipeline moves via workflow wipe opportunity custom fields. All migration is API-side.

      Never: delete before empty

      Deleting a GHL pipeline deletes its opportunities. Archive only after count = 0.

      Never: delete engine tags

      The dormant engine tag surface stays frozen — never renamed or deleted — so it can re-attach later.

      Never: write without a dry-run

      Every mutating script runs read-only first and emits a reviewable diff. No exceptions.
