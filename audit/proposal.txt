GHL Cleanup & Appointment Show-Rate
Optimization Sprint
Men's Wellness Centers · Richmond, Virginia Beach, Newport News
Objective
Rebuild and standardize the GoHighLevel CRM architecture for all three center locations, creating a
clean, documented, and scalable system that improves operational visibility and reduces
appointment no-shows.
The goal of this sprint is to establish a clean operating framework that allows the team to confidently
manage, maintain, and optimize the CRM moving forward.
Rather than modifying the live system directly, workflows will be audited, rebuilt, and tested in a
dedicated staging environment before migration into production.
Engine context: An AI engine is already live on top of GHL, handling webchat, SMS replies,
appointment booking, and dormant lead reactivation (Sniper). All GHL cleanup work must remain
compatible with the engine. Protected tags, custom fields, and pipeline stages are documented in
Phase 1 deliverables.
Sprint Duration
● Estimated duration: 4-5 weeks
● Consultant availability (Rosy Azpeitia, GHL Expert): 4 hours per day
● Client stakeholder (Eric O'Brien, COO): available during Week 1 for clarification, approvals, and
key decisions

Phase 1 — Discovery, Audit & Planning (Week 1)
Goal
Gain a complete, engine-aware understanding of the current CRM architecture before making any
changes. Every audit decision in this phase is made with the live AI engine in mind — preserving
what it depends on while identifying what can be cleaned, merged, or removed.
Deliverable 1 — System Inventory
Document all active CRM assets across all three locations:
● Workflows (all 133, not just the 24 published)
● Pipelines and opportunity stages
● Calendars
● Forms and surveys
● Tags (full taxonomy)
● Custom fields
● Triggers
● SMS and email templates
● Integrations
Deliverable 2 — Workflow Audit with Bot-Dependency Flag
For each of the 133 workflows, document:
● Purpose
● Trigger (tag / stage / contact / appointment)
● Actions and wait steps
● Status: Keep / Merge / Remove / Rebuild
● Bot-dependent: Yes / No / Unknown
The bot-dependency flag is the most critical column in this audit. The engine changes GHL state —
adding tags, moving pipeline stages, creating contacts, booking appointments — and those state
changes fire workflows. A workflow that looks redundant in isolation may be load-bearing for the
engine. Any workflow flagged Bot-dependent: Yes must be reviewed with Jean before it is modified
or removed.
Protected tags (do not rename or delete): src:bf-web · bf_test · chatbot_lead · concern_trt ·
concern_ed · concern_weight_loss · payment_insurance · payment_cash · location_rva ·
location_vba · location_npn · funnel_entry · funnel_consultation_booked · appt_confirmed ·

Deliverable 3 — Pipeline Stage Comparison & Canonical Taxonomy
Produce a side-by-side comparison of pipeline stage names across Richmond, Virginia Beach, and
Newport News. Identify any naming inconsistencies across locations, then propose a single
canonical stage taxonomy to be standardized in Phase 2.
This matters because the engine's stage-detection logic is location-agnostic. If Richmond and
Newport News use different names for the same stage, the bot misreads contacts that span
locations.
Deliverable 4 — TCPA / DND Gap Analysis
Audit the current opt-out and DND handling across all workflows:
● Which workflows check for DND or STOP status before sending
● Which workflows do not check (compliance risk)
● Whether the engine's STOP-reply handling (which halts the bot automatically) is consistent with
what GHL workflows do
● Gaps between bot behavior and GHL workflow behavior on opt-out
Deliverable 5 — Findings Report
Synthesize the audit into an actionable findings report covering:
Standard findings
● Duplicate workflows
● Conflicting automations
● Naming convention violations
● Messaging gaps
● Opportunity pipeline inconsistencies
Engine-split no-show risk points
No-show risks are categorized into two buckets to prevent rebuilding what the engine already
handles:
● Engine-owned: Already handled by engine — do not rebuild:
● Dormant lead reactivation (Sniper engine, live in production)
● Immediate no-show SMS via Sniper cohort scoring
● Workflow-owned: Require GHL workflow coverage — build in Phase 2:
● Booking confirmation cascade (T-24h, T-4h, T-1h)
● Unconfirmed appointment follow-up
● Intake-not-completed nudge sequence
● Same-day and next-day no-show recovery
● Missed call recovery (auto-SMS within 15 sec)

Deliverables

Staging + engine QA: The engine can be pointed at the staging subaccount via a single env var
swap (GHL_LOCATION_ID). Coordinate with Jean before Phase 4 testing to run the bot against
staging for a validation day.
Phase 3 — Messaging & Show-Rate Optimization (Week 3)
Goal
Implement the member communication strategy. All templates must pass the engine's brand voice
rules.
Brand voice rules (enforced by engine on every outbound message)
● No "free" — use "no-cost"
● No "patient" — use "member"
● No "clinic" — use "center"
● No em-dashes — use commas or periods
● No "Hey buddy" / "Hey guys" — use first name
● Plain URLs only — no markdown links (they don't render in SMS)
● No urgency theatre — no "limited time," "tonight only," "ASAP," or "hurry"
● Required in every substantive SMS: Men's Wellness + main line 866-344-4955
Deliverables
● SMS library — Booking Confirmation, Intake Requests, Intake Follow-Up, Appointment
Reminders, Reschedule Requests, No-Show Recovery (workflow-owned tier only)
● Email library — supporting sequences
● Two-way communication logic — reply-based automation where appropriate

Verify all automations with the engine as an active actor before deployment.

● Correct tagging
● Correct opportunity movement
● Correct notifications
● Correct messaging
